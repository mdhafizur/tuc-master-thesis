\chapter{Detailed Case Studies}
\label{appendix:case-studies}

This appendix documents five representative outcomes from the evaluation: true positive, false positive, false negative, partial repair, and multi-CWE detection. The cases explain how the measured metrics translate into practical IDE behavior.

\section{Case 1: True Positive with Effective Repair}
\label{sec:case-tp-good}

\subsection*{Scenario}
SQL injection in an Express route detected by \texttt{qwen3:8b} with RAG.

\subsection*{Original Vulnerable Code}
\begin{lstlisting}[caption={SQL injection vulnerability (CWE-89)}]
app.get('/user/:id', (req, res) => {
  const userId = req.params.id;
  const query = `SELECT * FROM users WHERE id = ${userId}`;
  db.query(query, (err, results) => {
    if (err) return res.status(500).send('Error');
    res.json(results[0]);
  });
});
\end{lstlisting}

\subsection*{Model Output (JSON)}
\begin{lstlisting}[caption={qwen3:8b + RAG output}]
[
  {
    "message": "SQL injection via direct query concatenation",
    "severity": "error",
    "cwe": "CWE-89",
    "lineStart": 3,
    "lineEnd": 3,
    "fixedCode": "const query = 'SELECT * FROM users WHERE id = ?';\ndb.query(query, [userId], ...)"
  }
]
\end{lstlisting}

\subsection*{Analysis}
This is a clean true positive with correct location and CWE label. The repair is minimal, uses parameterization, and preserves intent. In practice, this is an example of low-friction remediation that developers can apply immediately.

\section{Case 2: False Positive (Over-Sensitivity)}
\label{sec:case-fp}

\subsection*{Scenario}
Secure username sanitization flagged as vulnerable by \texttt{gemma3:4b} (LLM-only).

\subsection*{Original Secure Code}
\begin{lstlisting}[caption={Secure input validation (not vulnerable)}]
function sanitizeUsername(username) {
  const sanitized = username.replace(/[^a-zA-Z0-9_]/g, '');
  if (sanitized.length < 3 || sanitized.length > 20) {
    throw new Error('Username must be 3-20 characters');
  }
  return sanitized;
}
\end{lstlisting}

\subsection*{Model Output (JSON)}
\begin{lstlisting}[caption={False positive example}]
[
  {
    "message": "Potential code injection in replace()",
    "severity": "warning",
    "cwe": "CWE-94",
    "lineStart": 2,
    "lineEnd": 2
  }
]
\end{lstlisting}

\subsection*{Analysis}
This is a false positive: the whitelist regex and length checks are appropriate for the shown context. The CWE label is also mismatched. This case reflects the high-FPR behavior observed for several smaller configurations and illustrates triage overhead without security gain.

\section{Case 3: False Negative (Missed Vulnerability)}
\label{sec:case-fn}

\subsection*{Scenario}
Prototype pollution missed by \texttt{CodeLlama:latest} (LLM-only).

\subsection*{Original Vulnerable Code}
\begin{lstlisting}[caption={Prototype pollution (CWE-1321)}]
function mergeConfig(target, source) {
  for (let key in source) {
    target[key] = source[key];
  }
  return target;
}

const userConfig = JSON.parse(req.body.config);
mergeConfig(globalConfig, userConfig);
\end{lstlisting}

\subsection*{Model Output (JSON)}
\begin{lstlisting}[caption={Missed detection}]
[]
\end{lstlisting}

\subsection*{Analysis}
This is a false negative. The loop allows attacker-controlled keys such as \texttt{\_\_proto\_\_}, enabling object-prototype mutation. A robust output would flag CWE-1321 and suggest guarded property assignment (for example, \texttt{Object.prototype.hasOwnProperty.call(source, key)}). The case demonstrates recall limits for some model families without stronger grounding.

\section{Case 4: Correct Detection, Incomplete Repair}
\label{sec:case-repair-partial}

\subsection*{Scenario}
Path traversal is detected correctly, but the repair is only partial.

\subsection*{Original Vulnerable Code}
\begin{lstlisting}[caption={Path traversal (CWE-22)}]
app.get('/download', (req, res) => {
  const filename = req.query.file;
  const filepath = path.join(__dirname, 'uploads', filename);
  res.download(filepath);
});
\end{lstlisting}

\subsection*{Model Output (JSON)}
\begin{lstlisting}[caption={Detection with partial fix}]
[
  {
    "message": "Path traversal via user-controlled filename",
    "severity": "error",
    "cwe": "CWE-22",
    "lineStart": 3,
    "lineEnd": 3,
    "fixedCode": "const filename = path.basename(req.query.file);"
  }
]
\end{lstlisting}

\subsection*{Analysis}
Detection quality is good, but remediation is incomplete. Using \texttt{path.basename()} reduces traversal risk, yet authorization and allowlist checks are still needed to prevent unintended file access. This case shows why repairs remain advisory and require project-specific developer review.

\section{Case 5: Complex Multi-CWE Case}
\label{sec:case-multi-cwe}

\subsection*{Scenario}
Two vulnerabilities in one endpoint: hard-coded admin token and command injection.

\subsection*{Original Vulnerable Code}
\begin{lstlisting}[caption={Multiple vulnerabilities in one handler}]
app.post('/admin/exec', (req, res) => {
  const token = req.headers['x-auth-token'];
  if (token === 'admin123') { // CWE-798
    const cmd = req.body.command;
    exec(cmd, (err, stdout) => { // CWE-78
      if (err) return res.status(500).send(err.message);
      res.send(stdout);
    });
  } else {
    res.status(401).send('Unauthorized');
  }
});
\end{lstlisting}

\subsection*{Model Output (JSON)}
\begin{lstlisting}[caption={Two findings from one function}]
[
  { "cwe": "CWE-798", "lineStart": 3, "lineEnd": 3, "severity": "error" },
  { "cwe": "CWE-78",  "lineStart": 5, "lineEnd": 5, "severity": "error" }
]
\end{lstlisting}

\subsection*{Analysis}
Both issues are correctly identified and separated, which is useful for IDE diagnostics and prioritization. The credential fix is straightforward (move token to secure configuration). Command-execution repair remains design-sensitive and usually requires architectural change or strict command allowlists.
