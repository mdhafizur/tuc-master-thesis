\chapter{Privacy Verification Evidence}
\label{app:privacy-verification}

This appendix documents the evidence used to support Requirement R5 (privacy-preserving operation): network behavior, local boundary design, and configuration checks.

\section{Network Traffic Analysis}
\label{sec:privacy-network-analysis}

\subsection{Test Configuration}
Traffic was monitored during representative inline and audit workflows on the evaluation environment from Section~\ref{sec:eval-setup}:
\begin{itemize}
  \item macOS (arm64), VS Code extension development build
  \item Local backend (\texttt{localhost:3000})
  \item Ollama runtime (\texttt{localhost:11434})
  \item Curated vulnerable and secure samples
\end{itemize}

\subsection{Monitoring Method}
\begin{lstlisting}[language=bash, caption={External-traffic capture command}]
sudo tcpdump -i any -n 'not (host 127.0.0.1 or host ::1)' \
  -w /tmp/code-guardian-traffic.pcap
\end{lstlisting}

\subsection{Results}
\paragraph{Code analysis workflows.}
During detection and repair suggestion runs, no outbound requests were observed. Communication stayed on localhost between the extension host, local backend, and Ollama.

\paragraph{Knowledge refresh workflow.}
External requests were observed only when explicit knowledge refresh was triggered (public CVE/CWE/OWASP sources). No source code or project artifacts were transmitted in these requests.

\subsection{Out-of-Scope Threats}
The privacy boundary does not mitigate local host compromise, physical access attacks, or malicious third-party software on the same machine. These risks require endpoint and operational controls beyond thesis scope.

\section{Offline Operation Mode}

Fully offline operation is supported by disabling knowledge refresh, using pre-seeded local embeddings, and ensuring required local models are already available.
