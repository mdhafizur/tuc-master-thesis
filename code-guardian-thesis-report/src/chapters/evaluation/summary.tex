\section{Summary and Discussion}
\label{sec:eval-summary}

This chapter provides run-level evidence on three dimensions: detection quality, structured-output robustness, and latency. The evaluation procedure is documented in enough detail for reruns under comparable local conditions, but the reported values should be interpreted as descriptive evidence for this run rather than universal performance guarantees.

\subsection*{Key Findings}

\begin{itemize}
  \item \textbf{Best overall local configuration in this run:} \texttt{qwen3:8b} with RAG reached the highest F1 (63.76\%) with moderate secure-sample FPR (26.67\%).
  \item \textbf{RAG is model-dependent:} it improved \texttt{qwen3:8b} (58.12\% $\rightarrow$ 63.76\%) and \texttt{qwen3:4b} (54.90\% $\rightarrow$ 58.67\%), but degraded \texttt{gemma3:4b} and \texttt{CodeLlama:latest}.
  \item \textbf{False-positive control is the primary bottleneck:} most LLM configurations flagged all secure samples (FPR 100\%), which is costly for day-to-day IDE use.
  \item \textbf{SAST remains operationally valuable:} Semgrep had much lower recall (9.73\%) but low secure-sample FPR (6.67\%) and very low latency, making it a useful low-noise anchor.
  \item \textbf{Privacy objective is satisfied in the evaluated architecture:} code analysis stayed local; optional network activity is limited to public vulnerability metadata refresh.
\end{itemize}

Supplementary case-study material in the repository provides representative true-positive, false-positive, false-negative, and repair-quality examples that explain why these aggregate metrics matter in practice.

\subsection{False Positive Control and Practical Implications}
\label{sec:eval-fpr-discussion}

Secure-sample FPR is the strongest practical predictor of developer acceptance for IDE assistants. High FPR increases interruptions, triage cost, and alert fatigue.

\paragraph{Main causes of high FPR in this run.}
\begin{enumerate}
  \item \textbf{Prompt bias toward over-reporting:} models are encouraged to flag potential issues but are weakly incentivized to abstain.
  \item \textbf{Limited negative calibration:} models are not specifically tuned to output confident ``no issue'' decisions on secure snippets.
  \item \textbf{Strict detection scoring:} any emitted issue is counted as a positive finding, so speculative outputs inflate FPR.
  \item \textbf{Capacity/context limitations:} smaller or less stable models struggle to distinguish suspicious patterns from exploitable weaknesses.
\end{enumerate}

\paragraph{Hypothetical triage burden illustration.}
The following example is a sensitivity illustration, not a deployment forecast. For a project with 500 functions and an assumed 10\% vulnerable-code share, FPR 100\% implies roughly 450 false positives; at 2 minutes per alert, this is about 15 hours of review overhead. At FPR 26.67\%, expected false positives drop to about 120 (about 4 hours under the same assumption). Even this lower noise level remains expensive for always-on workflows.

\subsection{Hybrid SAST + LLM Integration Strategies}
\label{sec:eval-hybrid-sast}

The measured results motivate, but do not directly benchmark, a hybrid model in which SAST provides low-noise anchors and LLMs add contextual reasoning and repair suggestions. A pragmatic workflow pattern is:
\begin{enumerate}
  \item Run Semgrep/CodeQL to identify candidate locations.
  \item Apply LLM analysis only to those locations with focused context.
  \item Rank findings by combined provenance (SAST only, LLM only, both).
\end{enumerate}

This pattern is an engineering recommendation derived from the measured trade-offs of the individual tools: Semgrep contributes low secure-sample FPR and very low latency, while \texttt{qwen3:8b} contributes stronger contextual recall and repair-oriented output. Quantifying the combined profile requires a dedicated hybrid benchmark and is therefore left as future work rather than presented as a measured result here.

\subsection*{Operational Interpretation for Deployment}

Configuration should follow workflow phase rather than a single global default:
\begin{itemize}
  \item \textbf{Inline editing:} prioritize low FPR, parse stability, and predictable latency.
  \item \textbf{Pre-merge review:} use stronger models with higher recall and explicit manual triage.
  \item \textbf{Security audit:} combine deterministic and LLM findings, sorted by confidence/provenance.
\end{itemize}

\subsection*{Limitations}

\begin{itemize}
  \item The scored dataset is small (\texttt{n=128}; 113 vulnerable, 15 secure), so secure-sample FPR has wide uncertainty intervals.
  \item The benchmark is mostly snippet-centric; generalization to large multi-file repositories remains limited.
  \item R6 is evaluated primarily via latency and workflow observation; no controlled user study was run.
  \item Baseline normalization into a shared label schema can introduce taxonomy-mapping bias.
\end{itemize}

\paragraph{Statistical power considerations.}
The small secure-sample size ($n=15$) limits the precision of FPR estimates. With 15 samples, the 95\% Wilson confidence interval for an observed FPR of 26.67\% (4/15) spans [10.90\%, 51.95\%]—too wide for confident deployment decisions. To achieve a 95\% CI width of $\pm$10 percentage points around a true FPR of 25\%, approximately 75 secure samples would be required; for $\pm$5pp precision, approximately 300 samples. Future evaluation iterations should prioritize expanding the secure-sample set to enable statistically meaningful FPR comparisons across configurations.

\subsection*{Negative Results and Boundary Conditions}

\begin{itemize}
  \item RAG is not uniformly beneficial across models.
  \item Several configurations retain FPR 100\%, which is unsuitable for always-on inline use.
  \item Nearby model differences should be read as directional evidence, not inferential claims.
\end{itemize}
